# Kubernetes Deployment for Clawless on MicroK8s

## Context

Clawless currently deploys via `docker-compose.yml`. This plan adds Kubernetes manifests targeting a home lab MicroK8s 1.35 cluster. Clawless is an infrastructure service pinned to the primary node. External access is handled by an existing cloudflared tunnel — no ingress or local exposure needed, just a ClusterIP service. Manifests live in this repo; the user will symlink into the home_k8s repo.

## Cluster Conventions (from cluster-reference.md)

- MicroK8s v1.35, Calico CNI, Traefik ingress (not used here)
- Cloudflared already deployed in `cloudflared` namespace
- Storage: `microk8s-hostpath` StorageClass, `WaitForFirstConsumer`, `Delete` reclaim policy
- Node pinning: `nodeSelector: { role: primary }` for infrastructure services
- Convention: ClusterIP only, never LoadBalancer/NodePort
- Secrets never committed to git

## File Layout

```
deploy/k8s/
  namespace.yaml
  pvc.yaml
  secret-template.yaml   # committed — copy to secret.yaml and fill in
  secret.yaml            # gitignored — real secrets
  deployment.yaml
  service.yaml
  kustomization.yaml
  clawless.toml          # user edits this, kustomize generates ConfigMap from it
```

No ingress — cloudflared points directly to the ClusterIP service.
Secret uses a template pattern: `secret-template.yaml` is committed, user copies to `secret.yaml` and fills in values. `secret.yaml` is gitignored.

## Resources

### 1. Namespace (`namespace.yaml`)
- `clawless` namespace, per the one-namespace-per-app convention

### 2. PVC (`pvc.yaml`)
- Name: `clawless-home`
- StorageClass: `microk8s-hostpath`
- AccessMode: `ReadWriteOnce`
- Size: `5Gi`
- Single PVC with `subPath` mounts for all directories (`.claude`, `workspace`, `data`, `logs`, `plugin`)
- Rationale: app models everything as subdirectories of HOME; `clawless-init` scaffolds one tree

### 3. ConfigMap (generated by kustomize)
- Name: `clawless-config` (with `disableNameSuffixHash: true`)
- Source: local `clawless.toml` file in `deploy/k8s/`
- Mounted as a single file at `/home/clawless/clawless.toml` (read-only)

### 4. Secret (template pattern)
- `secret-template.yaml` — committed to git, contains placeholder values
- User copies to `secret.yaml`, fills in real values
- `secret.yaml` is added to `.gitignore`
- Contains `ANTHROPIC_API_KEY` (required) and optional Twilio creds
- Uses `stringData` for readability (Kubernetes base64-encodes on apply)
- Injected via `envFrom.secretRef`

### 5. Deployment (`deployment.yaml`)
- **Replicas**: 1 (SQLite single-writer constraint)
- **Strategy**: `Recreate` (no two pods running simultaneously)
- **Node pinning**: `nodeSelector: { role: primary }`
- **Security context**: `runAsUser: 1000`, `runAsGroup: 1000`, `fsGroup: 1000` (matches `clawless` user in Dockerfile)
- **Init container**: same image, runs `clawless-init /mnt/clawless-home` with full PVC mounted at `/mnt/clawless-home`. Idempotent — safe on every restart (uses `mkdir(exist_ok=True)`, only writes templates if missing)
- **Main container**:
  - Image: `ghcr.io/<owner>/clawless:latest` with `imagePullPolicy: Always`
  - Port: 18265
  - Command: `["/bin/bash", "-c", "clawless 2>&1 | tee /home/clawless/logs/clawless.log"]`
  - Env: `PORT=18265` + `envFrom: secretRef: clawless-secrets`
  - Liveness: `httpGet /health :18265`, initialDelay 30s, period 30s
  - Readiness: `httpGet /health :18265`, initialDelay 10s, period 10s
  - Resources: requests 512Mi/250m, limits 1Gi/1000m
- **Volume mounts** (main container):

  | Mount path | Source | Mode |
  |---|---|---|
  | `/home/clawless/.claude` | PVC subPath `.claude` | rw |
  | `/home/clawless/workspace` | PVC subPath `workspace` | rw |
  | `/home/clawless/data` | PVC subPath `data` | rw |
  | `/home/clawless/logs` | PVC subPath `logs` | rw |
  | `/home/clawless/plugin` | PVC subPath `plugin` | ro |
  | `/home/clawless/clawless.toml` | ConfigMap key `clawless.toml` | ro |

### 6. Service (`service.yaml`)
- Name: `clawless`
- Type: `ClusterIP`
- Port: 18265 -> targetPort 18265
- Selector: `app: clawless`

### 7. Kustomization (`kustomization.yaml`)
```yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: clawless
resources:
  - namespace.yaml
  - pvc.yaml
  - secret.yaml
  - deployment.yaml
  - service.yaml
configMapGenerator:
  - name: clawless-config
    files:
      - clawless.toml
    options:
      disableNameSuffixHash: true
```

## Container Image

Image is pulled from GHCR: `ghcr.io/<owner>/clawless:<tag>` (public, no pull secrets needed).
Deployment uses `imagePullPolicy: Always` for `:latest` or `IfNotPresent` for pinned tags.

### GitHub Action: Build and Push to GHCR (`.github/workflows/docker-publish.yaml`)

Uses the standard GitHub-recommended pattern for publishing container images:

- **Trigger**: push to `main` branch (and optionally on tags matching `v*`)
- **Permissions**: `packages: write`, `contents: read`
- **Steps**:
  1. Checkout
  2. Log in to GHCR via `docker/login-action` using `GITHUB_TOKEN` (no manual PAT needed)
  3. Extract metadata (tags, labels) via `docker/metadata-action`
     - Push to `main` -> tags image as `latest`
     - Push a `v*` tag -> tags image with that version
  4. Build and push via `docker/build-push-action` with layer caching

```yaml
# .github/workflows/docker-publish.yaml
name: Build and publish container image

on:
  push:
    branches: [main]
    tags: ["v*"]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - uses: docker/metadata-action@v5
        id: meta
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=raw,value=latest,enable={{is_default_branch}}
            type=semver,pattern={{version}}
            type=sha

      - uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

Note: The repo must have GitHub Packages enabled (it is by default). The first push makes the package private; go to the package settings on GitHub to make it public (so MicroK8s can pull without `imagePullSecrets`).

## Deployment Workflow

```bash
# 1. Ensure addons (likely already enabled)
microk8s enable dns hostpath-storage

# 2. Edit deploy/k8s/clawless.toml with channel config

# 3. Create secret from template
cp deploy/k8s/secret-template.yaml deploy/k8s/secret.yaml
# Edit deploy/k8s/secret.yaml with real API keys

# 4. Deploy
microk8s kubectl apply -k deploy/k8s/

# 5. Verify
microk8s kubectl -n clawless get pods -w
microk8s kubectl -n clawless logs deploy/clawless -f

# 6. Add tunnel route in Cloudflare dashboard
#    pointing to the clawless ClusterIP service on port 18265
```

## Update Workflow

```bash
# Code change -> GitHub Action pushes new image to GHCR, then:
microk8s kubectl -n clawless rollout restart deployment/clawless

# Config change -> edit deploy/k8s/clawless.toml, then:
microk8s kubectl apply -k deploy/k8s/
microk8s kubectl -n clawless rollout restart deployment/clawless
```

## Verification

1. `microk8s kubectl -n clawless get pods` — pod Running, 1/1 ready
2. `microk8s kubectl -n clawless logs deploy/clawless` — no errors, startup complete
3. `microk8s kubectl -n clawless exec deploy/clawless -- curl -s localhost:18265/health` — returns `{"status":"ok"}`
4. Check PVC bound: `microk8s kubectl -n clawless get pvc`
5. Verify init ran: `microk8s kubectl -n clawless exec deploy/clawless -- ls /home/clawless/data/`
